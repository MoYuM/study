大模型流式生成中途网络中断 / 用户刷新页面，要做到「不从头重生成、不重复输出」，关键在服务端记录生成进度，前端按断点续拉。

**整体思路**

- 每个会话/消息分配唯一 id；服务端边生成边把「已生成内容 + 进度（token 偏移 / chunk 序号）」写入缓存或库（Redis/DB）。
- 前端重连或刷新时带上 messageId + 已收到的偏移，请求「续传」接口。
- 服务端：若该消息已生成完 → 直接返回完整结果；若仍在生成 → 先补发「断点之前已生成但前端没收到」的部分，再继续推送新增量；前端按偏移去重，避免重复渲染。

**后端如何「继续生成」**

- 简单可靠：把 prompt + 已生成前缀拼回去，让模型从前缀继续 decode（多数云 API 只能这样「逻辑续写」）。
- 自建推理：保留该请求的 KV cache / 生成状态，真正从断点继续，省去重算前缀。
- 配合幂等：续传请求要幂等，避免重复触发计费/落库。

**Agent 长任务**

- 用任务状态机把每一步（规划 / 工具调用 / 中间结果）持久化；中断后从「最近一个成功步」恢复，而不是整条链重跑——这也是 Agent「可中断、可恢复、可人工接管」的基础。

**与文件断点续传的区别**：文件续传是按已上传字节分片续传（内容确定）；流式生成续传是按「已生成 token / 消息进度」续推，内容是动态产生的，还要处理推理状态与去重。
