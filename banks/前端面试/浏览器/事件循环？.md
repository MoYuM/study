事件循环是浏览器/Node 协调异步任务的机制，保证单线程 JS 非阻塞运行。注意：单线程 JS **没有「并行」**，它是**异步非阻塞**——遇到要等的活先挂一边、好了再回来，全程只有一个「柜员」在干活。

## 同步 vs 宏任务：差别就一个字——「现在」还是「以后」

| | 同步任务 | 宏任务 |
| --- | --- | --- |
| 何时跑 | **现在，立刻** | **以后，将来某一轮** |
| 在哪 | 直接在调用栈上跑 | 先存进队列排队 |
| 怎么跑 | 一行接一行连续跑完 | 一轮只取**一个**出来跑 |

**同步任务根本不排队**，它就是「正在一行行执行的普通代码」；宏/微任务是「被推迟、等同步跑完之后才跑的代码」。两者不在同一层面，不冲突；真正争先后的只是「以后」里的宏 vs 微。

## 执行顺序（核心主线）

同步代码执行清空调用栈 → 清空**全部**微任务队列 → 取**一个**宏任务执行 → **再清空全部微任务** → 如此循环。关键不对称：**微任务一次清空一整队，宏任务一次只取一个**；且每跑完一个宏任务都会再清一次微任务（宏任务里新产生的微任务，会在下一个宏任务之前被清掉）。渲染（如有）是事件循环中一个独立阶段（在清空微任务之后），本身不属于宏任务。

```
1. 同步代码全跑完（栈空）
2. 清空所有微任务
3.（可能渲染一帧）
4. 取一个宏任务跑
5. 回到第 2 步：再清空所有微任务
6. 再取下一个宏任务……
```

## 调用栈 / Web API / 回调队列 / 事件循环（四件套图）

```
        ┌─────────────────────────────┐
        │   你的 JS 代码 (一行行)       │
        └──────────────┬──────────────┘
                       │ 调用函数
                       ▼
   ┌───────────────────────────┐        ┌──────────────────┐
   │      调用栈 Call Stack      │ 调用→  │   Web APIs        │
   │  (同步任务在这里"现在"执行)   │        │ setTimeout/fetch  │
   │   main() → foo() → ...      │        │ DOM事件 (浏览器提供)│
   └───────────────┬───────────┘        └─────────┬────────┘
                   │                              │ 计时到/请求回来
       栈空了？ ◄──┤                              ▼
            │      │                   ┌────────────────────┐
            ▼      │                   │ 回调队列 Callback Q  │
   ┌────────────────────┐              │  (宏任务在这里排队)   │
   │  事件循环 Event Loop │ ◄─ 栈空时取队首 ┘
   │ "栈空了吗?空就搬一个" │
   └────────────────────┘
```

## 分类与坑

- **宏任务**：setTimeout、setInterval、I/O、MessageChannel。
- **微任务**：Promise.then、await 后面的代码、queueMicrotask、MutationObserver。
- 常见坑：`new Promise(fn)` 的 `fn` 是**同步执行**的，只有 `.then` 回调才是微任务；await 后面的代码会被包装成微任务；浏览器与 Node 的事件循环阶段划分不同（Node 有 timers/poll/check 等阶段）。

## 例题

```js
console.log(1);
setTimeout(() => console.log(2));
Promise.resolve().then(() => console.log(3));
Promise.resolve().then(() => console.log(4));
console.log(5);
// 输出 1,5,3,4,2 —— 同步(1,5) → 微任务整队(3,4) → 宏任务(2)
```
