HTTPS = HTTP + TLS。握手用**非对称加密协商出对称密钥**，之后用**对称加密传输**（兼顾安全与性能）。

## TLS 握手流程

```
客户端                              服务器
  │── ClientHello ────────────────►│  支持的TLS版本、加密套件、随机数
  │◄─ ServerHello + 证书 ──────────│  选定套件、随机数、CA证书
  │
  │  [客户端验证证书]
  │  └─ 用CA公钥验证证书签名 → 确认服务器身份（防中间人）
  │
  │── 交换密钥材料 ────────────────►│  RSA: 发加密的预主密钥
  │                                 │  ECDHE: 交换DH参数
  │
  │  [双方各自推导出相同的会话密钥（对称密钥）]
  │
  │── Finished ──────────────────►│  用会话密钥加密，互相验证
  │◄─ Finished ───────────────────│
  │
  │  [后续 HTTP 通信全用 AES 对称加密]
```

## 为什么两种加密混用

| 加密方式 | 特点 | 用途 |
|---------|------|------|
| 非对称（RSA/ECDHE） | 安全，但**慢** | 握手阶段协商密钥 |
| 对称（AES） | 快 | 加密实际传输内容 |

## CA 证书的作用

防止**中间人攻击**：客户端验证服务器证书由受信任 CA 签发，确认「我在和真正的服务器通信，而不是被劫持」。
