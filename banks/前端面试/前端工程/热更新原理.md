HMR（热模块替换）：开发时模块代码变更后，通过 WebSocket 通知浏览器，运行时只替换变更的模块，不刷新整页、保留应用状态。

- Webpack-dev-server：监听文件变化 → 编译出补丁（manifest + 更新 chunk）→ 通过 WS 推送 → 客户端 runtime 用 `module.hot.accept` 替换模块。
- Vite：基于浏览器原生 ESM，仅让浏览器重新请求变更的模块，无需打包，更新更细、速度更快。

## 为什么必须用 WebSocket（不是 HTTP）

HMR 需要**服务端主动推送**「某模块更新了」的通知，而 HTTP 只能浏览器发起、服务端被动应答（要主动推只能轮询）。WebSocket 是全双工长连接，服务端可随时主动推，正契合此场景。

## 核心机制：`module.hot.accept` + 冒泡（面试高区分度，常被问漏）

浏览器拿到新代码后，光下载还不够，靠 HMR runtime 完成「局部替换不刷新整页」：

1. **谁接收更新**：一个模块能被热替换，前提是它**自己 `module.hot.accept`**，或它的**某个父模块** accept 了它。
2. **冒泡（bubbling up）**：若改动模块 C 自己没 accept，runtime 往上问「import 了 C 的父模块 B 接不接」，B 不接再往上……一路冒泡直到某层 accept；**若冒泡到入口（entry）都没人接 → HMR 失败，退化为整页刷新**。（这就是「模块不支持局部替换会怎样」的答案。）
3. **替换过程**：所有失效模块先调 `dispose` handler（清理旧副作用、保存要留的状态）→ 卸载旧模块 → 更新 hash → 调用 `accept` handler 用新模块重新执行。
4. **状态为何能保住**：整页没刷新，全局/闭包里的运行时状态本就还在；React/Vue 的热更新插件（如 react-refresh）在 accept 里只替换组件渲染逻辑、**保留组件 state**，故数据不丢。

> 一句话：单个 accept handler 可兜住整棵子模块树的更新——没写 HMR 代码的模块就靠冒泡交给上层处理。
